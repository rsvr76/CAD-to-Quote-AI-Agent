# Step 3 — Agent Inputs via Chat (Confirm + Fill Gaps)

**Phase:** 2 — User Input Collection  
**Input:** Auto-extracted geometry (from Step 2)  
**Output:** Confirmed manufacturing parameters

---

## Goal

Engineer confirms or enters manufacturing information that may not be guaranteed even with STEP. Step 2 provides auto-detected *estimates* (holes/pockets/depth) which this step confirms; if changed, the UI records it as an **Engineer override** (optionally with a short note).

---

## Inputs Collected

| Input | Type | Reason |
|---|---|---|
| `num_holes` | int | STEP can detect cylinders, but engineer confirms (slots/counterbores can confuse) |
| `num_pockets` | int | STEP can detect concave pockets, but engineer confirms intent (cosmetic vs functional) |
| `max_depth_mm` | float | **Max drilling depth among all holes**; confirm for drilling cycle-time accuracy |
| `material` | enum | Determines machinability factor |
| `quantity` | int | Setup cost scaling |
| `tolerance_class` | enum | Determines finishing operations |

---

## Auto-Detected Prefill + Confidence (from Step 2)

Step 2 can provide best-effort estimates to prefill the chat (and to reduce typing), along with a simple confidence label (low/medium/high) so the engineer knows where to double-check:

- `num_holes_est`, `num_holes_confidence`
- `num_pockets_est`, `num_pockets_confidence`
- `max_depth_mm_est`, `max_depth_confidence`

These estimates are advisory only. If the engineer edits them, the UI records an **Engineer override**.

---

## Material Options

| Material | Machinability Factor |
|---|---|
| Aluminium | 1.0 (baseline) |
| Steel | 1.8 |
| Brass | 1.2 |
| Titanium | 5.0 |
| ABS | 0.6 |

### Where machinability is used

- The ML model receives `material` as an encoded feature (so material choice directly affects the predicted machining time).
- The **machinability factor** is primarily used during **synthetic dataset generation** to simulate real-world differences in machining difficulty across materials (so the trained model learns the correct trend even when trained on synthetic data).

---

## Tolerance Options

| Class | Description |
|---|---|
| Standard | No grinding required |
| Fine | Grinding finishing pass added |
| Ultra-Fine | Extended grinding, tight tolerances |

### Typical numeric meaning (guidance)

These are rough, shop-dependent guidelines (not guarantees), included to make the UI definitions concrete:

| Class | Typical tolerance (illustrative) |
|---|---|
| Standard | ~±0.5 mm |
| Fine | ~±0.1 mm |
| Ultra-Fine | ~±0.01 mm |

Downstream, we still treat this as a discrete `tolerance_class` input for routing + time prediction.

---

## Input Validation (Guardrails)

To prevent invalid quotes, the UI validates user inputs:

| Field | Validation |
|---|---|
| `num_holes` | integer, ≥ 0 |
| `num_pockets` | integer, ≥ 0 |
| `max_depth_mm` | > 0 |
| `quantity` | integer, ≥ 1 |
| `material` | one of allowed enum values |
| `tolerance_class` | one of allowed enum values |

---

## Example Chat Interaction

```
System: I detected 6 holes (confidence: medium) and 2 pockets (confidence: low).
    These typically require drilling and milling operations.
    Please confirm before I estimate machining time.

System: Confirm hole count? (Detected: 6)
Engineer: Yes, 6.

System: Confirm pocket count? (Detected: 2)
Engineer: Yes, 2.

System: I estimated max drilling depth among holes ≈ 25 mm. Confirm or enter the correct depth.
Engineer: 25 mm.

System: Select material: [Aluminium, Steel, Brass, Titanium, ABS]
Engineer: Steel

System: Quantity needed?
Engineer: 50

System: Tolerance class: [Standard, Fine, Ultra-Fine]
Engineer: Fine
```

---

## Key Principle

User input **overrides** any auto-detected estimate. If the engineer changes an auto-detected estimate, the UI records it as an **Engineer override**.

---

## Output

A confirmed parameter dictionary:

```python
user_inputs = {
    "num_holes": 6,
    "num_pockets": 2,
    "max_depth_mm": 25.0,
    "material": "Steel",
    "quantity": 50,
    "tolerance_class": "Fine",
}
```

These feed into Process Routing (Step 4) and Feature Vector Assembly (Step 7).

> These confirmed parameters form the **manufacturing intent layer** of the system and serve as inputs to both the **process routing engine (Step 4)** and the **machining time prediction model (via Step 7)**.
