# Step 4 — Process Routing

**Phase:** 3 — Cost Engine  
**Input:** Geometry features (Step 2) + User inputs (Step 3)  
**Output:** Final process routing + machine selections (with override metadata)

---

## Goal

Determine which machining processes are required for this part. Routing is entirely rule-based — no ML involved.

This step has two layers:

1) **Process routing** (what operations are needed) — deterministic rules
2) **Machine assignment** (which machine performs each active operation) — UI selection / configuration

---

## Routing Rules

| Process | Condition | Rationale |
|---|---|---|
| Turning | `aspect_ratio > 2.0` | Elongated parts need lathe operations |
| Milling | `num_pockets > 0` OR `surface_area_cm2 > 200` | Pockets require milling; large planar faces often require face-milling even without explicit pockets |
| Drilling | `num_holes > 0` | Any holes require drill operations |
| Grinding | `tolerance_class ∈ {Fine, Ultra-Fine}` | Hackathon simplification: treat “grinding” as a precision finishing operation for tight tolerance parts |

**Note on the `surface_area_cm2 > 200` threshold:** this is a pragmatic heuristic for hackathon scope (a tunable parameter). The intent is to catch parts with large flat faces that typically require facing/finishing passes even if no explicit pockets are detected.

---

## Implementation

```python
def route_processes(aspect_ratio, num_holes, num_pockets, 
                    surface_area_cm2, tolerance_class) -> dict:
    """Determine active machining processes based on rules."""
    return {
        "is_turning":  int(aspect_ratio > 2.0),
        "is_milling":  int(num_pockets > 0 or surface_area_cm2 > 200),
        "is_drilling": int(num_holes > 0),
        "is_grinding": int(tolerance_class in ("Fine", "Ultra-Fine")),
    }
```

## Display to Engineer

The routing decision is shown in the UI with the **reason** next to each process. This is the "Routing Explanation" layer — one of the three separate explainability layers described in the strategy.

The UI shows:

- The system’s **auto-detected processes** with the reason for each
- The list of **machines that can execute each process** (preselect a sensible default)

The engineer can **override** both:

- The process routing (toggle processes ON/OFF)
- The selected machine per active process

Example UI behavior:

```
Milling:   ON  (2 pockets detected)   [Change]
  Machine: 3-Axis VMC (dropdown: 3-Axis VMC / 4-Axis / 5-Axis)

Turning:   OFF (not elongated)        [Change]
  Machine: CNC Lathe (dropdown: CNC Lathe / Manual Lathe)
```

If the engineer changes anything, the dashboard marks it as an **Engineer override** and stores the override reason/note.

---

## Contradictions / Validation (Guardrails)

The UI checks for contradictions between confirmed inputs and routing selections.

Examples:

- `num_holes > 0` but engineer turns **Drilling OFF**
- `tolerance_class ∈ {Fine, Ultra-Fine}` but engineer turns **Grinding OFF**

In these cases, the UI warns and requires an override note (e.g., “holes will be drilled manually / purchased as pre-drilled”, “finishing handled externally”). The system always uses the **final confirmed** routing, but keeps this audit trail for trust.

---

## Output

Two sets of routing flags:

- **Auto routing** (rule output) — shown with reasons for transparency
- **Final routing** (user-confirmed) — used downstream

> The ML model always consumes the **final routing flags after engineer confirmation** (`routing_final`), ensuring predictions reflect the actual planned manufacturing process.

Final routing is four binary flags used downstream:

- As ML input features in the feature vector (Step 7)
- For process-specific cost attribution in explainability (Step 11)

Optionally, the UI also captures `machine_selection_final` (per active process). This is a UI/config input (not ML) that parameterizes per-machine hourly rates in Step 10.

Recommended output object shape (grouped for clarity):

```json
{
  "routing_decision": {
    "routing_auto":  {"is_turning": 0, "is_milling": 1, "is_drilling": 1, "is_grinding": 0},
    "routing_final": {"is_turning": 0, "is_milling": 1, "is_drilling": 1, "is_grinding": 0}
  },
  "machine_assignment": {
    "machine_selection_final": {
      "milling": "3-Axis VMC",
      "drilling": "VMC Drill"
    }
  },
  "overrides": {
    "engineer_override": true,
    "override_note": "Use 5-axis for pocket access",
    "contradictions_acknowledged": false
  }
}
```
