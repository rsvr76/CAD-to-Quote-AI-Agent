# Step 7 - Feature Vector Assembly

**Phase:** 3 - Cost Engine
**Input:** Geometry (Step 2) + User inputs (Step 3) + Routing flags (Step 4)
**Output:** 17-feature ML input vector

---

## Goal

Assemble all inputs into the feature vector that the Random Forest model expects. This is the single point where geometry, user inputs, routing flags, and derived features merge.

Critical requirement:

> This feature vector exactly matches the schema used during model training (same feature names + fixed ordering), ensuring consistent inference behavior.

---

## Feature Groups

### Geometry (from Step 2)

| Feature | Source |
|---|---|
| surface_area_cm2 | STEP B-Rep surface properties (OpenCascade) |
| volume_cm3 | STEP B-Rep volume properties (OpenCascade) |
| aspect_ratio | bounding box ratio |
| sv_ratio | surface_area / volume |
| slenderness | aspect_ratio / volume^(1/3) |
| pocket_density_proxy | num_pockets / (num_pockets + 1) (bounded pocket-density proxy) |

### User Inputs (from Step 3)

| Feature | Source |
|---|---|
| num_holes | engineer input |
| num_pockets | engineer input |
| max_depth_mm | engineer input |
| material | engineer input (label-encoded category for hackathon scope) |
| tolerance_class | engineer input (encoded: Standard=0, Fine=1, Ultra-Fine=2) |

### Routing Flags (from Step 4)

| Feature | Source |
|---|---|
| is_turning | rule-based (can be changed in UI; recorded as **Engineer override**) |
| is_milling | rule-based (can be changed in UI; recorded as **Engineer override**) |
| is_drilling | rule-based (can be changed in UI; recorded as **Engineer override**) |
| is_grinding | rule-based (can be changed in UI; recorded as **Engineer override**) |

### Derived Features

| Feature | Formula |
|---|---|
| complexity_score | (num_holes x 2) + (num_pockets x 3) + (surface_area / volume) |
| drilling_work | num_holes x max_depth_mm x is_drilling |

## Implementation

```python
MATERIAL_ENCODE = {"Aluminium": 0, "Steel": 1, "Brass": 2, "Titanium": 3, "ABS": 4}
TOLERANCE_ENCODE = {"Standard": 0, "Fine": 1, "Ultra-Fine": 2}

# IMPORTANT: inference must use the exact same feature order as training.
FEATURE_ORDER = [
    "surface_area_cm2",
    "volume_cm3",
    "aspect_ratio",
    "sv_ratio",
    "slenderness",
    "pocket_density_proxy",
    "num_holes",
    "num_pockets",
    "max_depth_mm",
    "material",
    "tolerance_class",
    "is_turning",
    "is_milling",
    "is_drilling",
    "is_grinding",
    "complexity_score",
    "drilling_work",
]

def validate_inputs(geometry, user_inputs, routing) -> None:
    if user_inputs["num_holes"] < 0:
        raise ValueError("num_holes must be >= 0")
    if user_inputs["num_pockets"] < 0:
        raise ValueError("num_pockets must be >= 0")
    if user_inputs["max_depth_mm"] <= 0:
        raise ValueError("max_depth_mm must be > 0")
    for key in ("surface_area_cm2", "volume_cm3", "aspect_ratio", "sv_ratio", "slenderness"):
        if geometry[key] <= 0:
            raise ValueError(f"{key} must be > 0")
    for key in ("is_turning", "is_milling", "is_drilling", "is_grinding"):
        if routing[key] not in (0, 1):
            raise ValueError(f"{key} must be 0 or 1")

def assemble_features(geometry, user_inputs, routing):
    validate_inputs(geometry, user_inputs, routing)

    mat_code = MATERIAL_ENCODE[user_inputs["material"]]
    tol_code = TOLERANCE_ENCODE[user_inputs["tolerance_class"]]

    # Bounded proxy (0..1) for “how pocket-heavy” a part is.
    pocket_density_proxy = user_inputs["num_pockets"] / (user_inputs["num_pockets"] + 1)
    drilling_work = (user_inputs["num_holes"] * user_inputs["max_depth_mm"]
                     * routing["is_drilling"])
    complexity_score = (user_inputs["num_holes"] * 2
                       + user_inputs["num_pockets"] * 3
                       + geometry["sv_ratio"])

    features = {
        "surface_area_cm2": geometry["surface_area_cm2"],
        "volume_cm3": geometry["volume_cm3"],
        "aspect_ratio": geometry["aspect_ratio"],
        "sv_ratio": geometry["sv_ratio"],
        "slenderness": geometry["slenderness"],
        "pocket_density_proxy": round(pocket_density_proxy, 4),
        "num_holes": user_inputs["num_holes"],
        "num_pockets": user_inputs["num_pockets"],
        "max_depth_mm": user_inputs["max_depth_mm"],
        "material": mat_code,
        "tolerance_class": tol_code,
        "is_turning": routing["is_turning"],
        "is_milling": routing["is_milling"],
        "is_drilling": routing["is_drilling"],
        "is_grinding": routing["is_grinding"],
        "complexity_score": round(complexity_score, 4),
        "drilling_work": round(drilling_work, 2),
    }

    # Safe model input (same order as training)
    X_row = [features[name] for name in FEATURE_ORDER]
    return features, X_row
```

Encoding note (materials): we use label encoding for hackathon scope. Tree models (Random Forest) can handle such categorical codes reasonably well because splits are threshold-based; for a production system, one-hot encoding is often preferred.

Scaling note: Random Forest does not require feature normalization. Tree-based models are generally insensitive to feature scale differences because they split on thresholds rather than distances.
